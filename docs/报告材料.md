**任务书原文（直接复制这段到报告里就可以）：**

> 选题二：MBTI 测试系统
>
> 主线任务：实现包含题库、测试交互、分数计算的系统。需通过特定算法计算出人格类型并输出详细解析。**强烈鼓励自创测评体系（如SBTI），定义全新的维度、权重，并在报告中详细阐述。**

> 额外挑战：使用图形化界面（如图片选择、情景模拟）进行交互，并在测试结束后使用 Echarts 生成包含交互图表的个人专属报告。

**我们的选题理解（写报告时用自己的话转述）：**

我们选的是题目二。但不想做那种"丢给你一套题，全答完才出结果"的传统问卷。我们想做一个不一样的——你每道题的选择会影响你接下来看到什么题，就像走迷宫一样。同时我们也想加一些传统 MBTI 没有的东西，比如独立于人格判定的心理倾向分析（这个在后面五维图那里会讲）。

【写报告时注意：这里就说"我们想做一个不一样的MBTI"，不要说"本系统创新性地采用二分树架构"——那是设计部分的内容。】

### 2. 需求分析

我们需要以下几个模块：

**模块一：题库**

要解决的问题：题目要能分开存放，不能写在代码里。不然每次改题都要改代码，太麻烦。

怎么做的：把所有题目放在一个 JSON 文件里，后端启动时读这个文件。每道题包含：题号、题目文字、四个选项。每个选项包含：文字、路由方向（L/R）、五维权重（5个数字）。

用到什么技术：JSON 文件读写。Python 自带的 json 库就能处理，不需要额外学东西。

可行性：完全可行，JSON 是标准格式。

**模块二：动态路由引擎**

要解决的问题：用户的选择要影响后续的题目走向，不是统一算分。

怎么做的：设计了一个"题池"的概念。每个题池有 5 道题和一个路由表。比如路由表写成 {"L": 2, "R": 3}，表示用户这 5 题答完之后，选 L 多就去 2 号池，选 R 多就去 3 号池。一层层走下去，4 层之后到达终点。

用到什么技术：字典查找、条件判断。大一学的 if-else 就够用。

可行性：完全可行，逻辑就是统计次数比大小。

**模块三：五维权重累加器**

要解决的问题：想在人格类型之外给用户一个更深层的心理倾向分析。但如果直接问用户"你上网是为了学知识还是为了娱乐"，那用户肯定会往好听的方向选。

怎么做的：把 5 个心理维度的权重藏在每个选项背后。用户选 A 还是选 D，路由方向可能一样，但权重加分不一样。用户全程不知道自己在被"测心"，最后累加出来的雷达图反映的是潜意识的倾向。

用到什么技术：数组加法。

可行性：完全可行。

**模块四：前端交互**

要解决的问题：用户需要一个好看的界面来答题和看结果。

怎么做的：用了 Vue 3 做前端。三个页面：首页（项目介绍+开始按钮）、答题页（动态加载题目）、结果页（显示人格+雷达图）。

用到什么技术：Vue 3、Vue Router、Axios（发 HTTP 请求）、ECharts（画雷达图）。

可行性：Vue 我们原来没学过，这部分主要靠 AI 生成和调试。ECharts 是现成的图表库，接入不难。

【写报告时注意：这里技术可行性可以写得朴实一点，比如"Vue之前没学过，现学的"、"ECharts看了看文档就能用"。】

**模块五：结果展示**

要解决的问题：结果不能只给四个字母，要好看、要有分析。

怎么做的：人格类型 + 五维雷达图 + 16 种人格 × 16 种配色。结果页会显示人格字母（大号渐变）、标题、描述、五维雷达图。

用到什么技术：ECharts 雷达图、CSS 渐变。

```
Step 11: 前端显示人格类型 + ECharts 五维雷达图
```

### 2. 详细设计

**核心算法一：二分树多数决路由**

这是整个系统最核心的算法，说人话就是：

> 每一组 5 道题，每道题有 4 个选项。这 4 个选项表面上只是不同的选择，但底下标注了方向——A 和 B 指向左，C 和 D 指向右。
>
> 5 题答完之后，统计一下选了左方向多少次、右方向多少次。哪边多就往哪边走。
>
> 比如说 5 题里选了 3 次左、2 次右，那就往左走，进入左分支的下一个题池。反之亦然。
>
> 这样一共走 4 层，最终到达 16 个人格类型中的某一个。

算法的优点和缺点：

| 优点 | 缺点 |
|------|------|
| 逻辑简单，不会算错 | 如果刚好 3:2 或 2:3，那少数的那两题完全没影响 |
| 容易实现，if-else 就行 | 极端情况下（刚好平局）需要有个默认方向 |
| 用户答完立刻出结果，不需要额外计算 | — |

对于它的缺点，我们觉得问题不大——人格测试本来就是看主要倾向，小偏差忽略不影响最终结果。而且每池 5 题是奇数，不会出现平局。

**核心算法二：五维权重累加**

这个跟路由是同时发生的，但是独立计算的。

每道题的每个选项背后有个 `weights` 数组，长度 5，每个数字 0-2，代表这道题在这个选项上对 5 个心理维度的影响力度。

> 每次用户选了一个选项，系统就把这 5 个数字加到 session 的累加器里。
>
> 用户全程不知道这件事。ta 以为自己在做"选 A 还是选 D"的普通选择题，实际上每选一次都在给五个心理维度默默加分。
>
> 20 题答完之后，累加器里就有了这个人在"精神续航、社交磁场、秩序洁癖、压力解压阀、脑力劳作"这五个维度上的累计得分。归一化之后就是五维雷达图。

这个算法的好处是：用户没法作弊——因为ta根本不知道有这个系统在跑。坏处是：需要对每个选项的权重做合理设计，不然累加出来的数据没意义。

**为什么不同路径不用相同的题池？**

（这个点你在报告里可以展开写一下，因为它是我们设计方案的核心特色）

如果让所有人都用同一套题来测"参与还是观察"，会出现一个问题：一个喜欢在群里聊天的人和一个喜欢默默潜水的人，面对同一道题时的心理感受完全不同。前者会觉得"这题说的就是我"，后者会觉得"这题跟我有什么关系"，然后随便选一个。

所以我们在 Stage 2 和后面的阶段，对已经分出路径的人用了不同的题池。比如：

- 已经判定为"外放型"的人，ta 在 Stage 2 遇到的题是关于"你在群聊中怎么参与"的
- 已经判定为"内收型"的人，ta 在 Stage 2 遇到的题是关于"你对自己的圈子有没有归属感"的

两道题测的是同一个东西（参与还是观察），但场景完全不一样。这样测出来的结果才准。

这叫"纵向耦合"——路径决定了你看到什么题。但"横向解耦"——同一层不同池的判定标准是一样的。

## 三、实现

### 1. 后端 API（app.py）

后端一共 4 个接口，都是用 Flask 写的。

**获取题池接口：**

```python
@app.route("/api/pool/<int:pool_id>")
def get_pool(pool_id):
    questions = engine.get_pool_questions(pool_id)
    return jsonify(questions)
```

这个接口做的事：根据题池编号，从题库 JSON 里读出对应的 5 道题，过滤掉权重和路由信息后返回给前端。前端拿到的是纯文本的题目和选项，看不到任何隐藏数据。

**提交答案接口：**

```python
@app.route("/api/pool/<int:pool_id>/submit", methods=["POST"])
def submit_pool(pool_id):
    data = request.get_json()
    answers = data["answers"]
    result = engine.submit_pool_answers(sess, pool_id, answers)
    return jsonify(result)
```

这个接口做的事：接收前端传过来的 5 个答案（比如 ["A","B","A","B","A"]），交给路由引擎处理。引擎统计方向、累加权重、查路由表，然后返回下一步去哪。

**生成对应代码时的提示词：**

> "帮我写一个 Flask 后端，有 4 个接口。第一个获取题池题目，第二个提交 5 个答案并路由，第三个创建新会话，第四个最终结算。题目存在 JSON 文件里。"

### 2. 路由引擎（engine.py）

**路由判定的核心代码：**

```python
def submit_pool_answers(self, session, pool_id, answers):
    pool = self.pools[str(pool_id)]
    questions = pool["questions"]

    route_counts = {"L": 0, "R": 0}
    for i, ans in enumerate(answers):
        opt = questions[i]["options"][ans.upper()]
        label = opt["route"]
        route_counts[label] += 1
        # 累加五维权重
        for j in range(5):
            session["weights"][j] += opt["weights"][j]

    # 多数决
    route_result = "L" if route_counts["L"] >= route_counts["R"] else "R"

    # 查路由表决定下一题池
    route_map = pool.get("route_map", {})
    next_pool_str = route_map.get(route_result)

    # Stage 4 的 route_map 指向 "D" 或 "F"（终局）
    if next_pool_str in ("D", "F"):
        session["done"] = True
        next_pool = None
    else:
        next_pool = next_pool_str

    return {"next_pool": next_pool, "is_final": session["done"], "route_result": route_result}
```

这段代码在做的事：遍历 5 个答案，每个答案查一下它对应的选项数据，取出 `route` 字段（L 或 R）来计数，取出 `weights` 数组来累加。5 个都处理完之后，比较 L 和 R 哪个多。多的那个方向就是本阶段的结果。然后根据这个结果查路由表，决定下一步去哪。

**人格生成的代码：**

```python
DIM_LETTERS = [
    {"L": "I", "R": "R"},  # Stage 1 → I(外放) / R(内收)
    {"L": "P", "R": "S"},  # Stage 2 → P(参与) / S(观察)
    {"L": "A", "R": "C"},  # Stage 3 → A(分析) / C(语境)
    {"L": "D", "R": "F"},  # Stage 4 → D(深度) / F(碎片)
]

# 把路径 ["L","R","L","L"] 转成字母 ["I","S","A","D"]
letters = []
for i, direction in enumerate(path):
    letters.append(DIM_LETTERS[i][direction])

type_code = "".join(letters)  # "ISAD"
```

这段代码做的事：路径里存的是 ["L","R","L","L"] 这样的方向数据，不能直接当人格码。

人格码应该是 "IPAD"、"RSCF" 这种字母组合。所以需要一个映射表——"第 1 层的 L 对应 I，第 2 层的 R 对应 S"，以此类推。映射完之后四个字母拼在一起就是人格码，再到人格描述表里查出标题和描述。

**生成对应代码时的提示词：**

> "现在我要改成二分树路由。每个题池 5 道题，答完之后根据 L/R 多数决去下一个池。每个选项有 route 和 weights 两个隐藏字段。路由走 route，权重累加独立进行。"

### 4. ECharts 雷达图组件

结果页使用 ECharts 画五维雷达图。

```javascript
// 核心配置
chartInstance.setOption({
    radar: {
        indicator: indicators.map(item => ({ name: item.name, max: 100 })),
    },
    series: [{
        type: 'radar',

```

这段代码做的事：ECharts 的雷达图需要两个数据——轴（indicators）和值（value）。轴就是 5 个维度的名称，值就是归一化后的 0-100 分数。把这两个数据丢进去，ECharts 自动画出一个五边形雷达图，每个轴的顶点连成一个面。

### 遇到的问题和解决方法

**问题 1：中文引号导致编译错误**

在写人格描述的时候，我们在文字里用了中文双引号，比如"是团队里的后勤保障"。结果 Python 把这个双引号当成了字符串的结束标志，导致编译报错，一连串报了 82 个错误。

解决：把所有中文双引号改成了「」角括号。后来我们约定，以后在代码里如果要引用中文内容，统一用角括号。

**问题 2：前后端联调的跨域问题**

前端和后端跑在不同的端口（前端 3000，后端 8080），浏览器会阻止这种跨域请求。刚开始前端一直调不通后端的接口，还以为代码写错了。

解决：在后端加了 CORS 配置，允许跨域访问。同时在前端的 Vite 配置文件里配了代理，开发时 /api 开头的请求自动转发到后端。

**问题 3：题库选项倾向太明显**

初版的题目，有些题目的选项设计得不好——A 选项明显是个好的方向，D 选项明显是个不好的方向。用户一眼就能看出"我应该选什么"，这样测出来的结果就不准了。

解决：逐池审查了所有 75 道题，把倾向太明显的选项重新设计。确保每个题目的四个选项都是中性的两难选择，用户需要认真思考才能做出决定。

### 关于 AI 辅助的说明

**本系统使用了以下 AI 平台和模型：**
- DeepSeek Chat（DeepSeek V4）：代码生成、架构讨论、调试辅助
- Gemini（Gemini 2.5 Pro）：题库设计、人格类型定义

**使用方式：** Chat 模式。AI 仅提供代码建议和设计方案，所有代码都经过人工审查和理解。

**没有使用：** Agent/自动编程模式、一键生成全量代码。

**具体哪些部分由 AI 辅助：**
1. 后端 Flask 框架的骨架代码（路由、接口定义）
2. 前端 Vue 3 的页面结构和组件（我们没有学过 Vue，这部分主要由 AI 生成后修改）
3. ECharts 雷达图的配置代码
4. 题库和人格描述的内容生成（由 Gemini 分池逐批生成，我们逐批审阅修改）

**哪些部分是自己完成的：**
1. 系统的架构设计（二分树路由 + 横向解耦纵向耦合）
2. 四个维度的定义和 16 种人格的命名
3. 题库的审校和修改
4. 前端页面的样式适配和调试
5. 将 AI 生成的各个模块拼接到一起并调试运行
