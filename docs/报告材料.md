## 一、构思

### 1. 项目背景和任务目标

**（直接抄任务书原文的选题二主线任务和额外挑战）**

> **主线任务：** 实现包含题库、测试交互、分数计算的系统。需通过特定算法计算出人格类型并输出详细解析。**强烈鼓励自创测评体系（如 SBTI），定义全新的维度、权重。**
>
> **额外挑战：** 使用图形化界面进行交互，并在测试结束后使用 Echarts 生成包含交互图表的个人专属报告。

我们选了这个题，但不想做传统的"一口气答完题再出结果"那种模式。我们想做一个用户每走一步都会影响后面走向的系统——就像走迷宫一样。同时我们自创了一套测评体系叫 NPFJ（Network Personality Factor Indicator），设计了 4 个全新维度和 16 种人格类型。

### 2. 需求分析

要实现这个目标，需要以下模块：

**【模块一：题库模块】**
题目不能写在代码里，不然改一题就要改代码。我们把所有题目放在一个 JSON 文件里，后端启动时读取。每道题包含题号、文字、4 个选项，每个选项背后有路由方向和 5 维权重。
- 解决了什么问题：题目和代码分离，改题只需要编辑 JSON，不用动代码
- 可行性：JSON 是标准格式，Python 自带 json 库

**【模块二：路由引擎模块】**
要实现"选什么就走到什么分支"，需要一套路由逻辑。我们设计了二分树结构：15 个题池（1+2+4+8），每池 5 题。用户答完 5 题后，系统统计 A/B方向 和 C/D方向 哪个多，多的决定走向。路由路径记录在会话里，最终 4 步拼出人格码。
- 解决了什么问题：实现了动态分流的测试流程，每个人的题目路径都不一样
- 可行性：统计次数 + 查路由表，大一水平的 if-else 就够

**【模块三：权重计算模块】**
路由决定的是人格类型，但我们还想在人格之外给出更深层的心理分析。我们在每个选项背后藏了 5 个维度的权重值（精神续航、社交磁场、秩序洁癖、压力解压阀、脑力劳作），用户答题时这些权重独立累加，最终生成五维雷达图。
- 解决了什么问题：人格判定和五维图完全解耦，互不影响，但结合起来能给出更立体的分析
- 可行性：数组加法，完全没有技术难度。整个计算器独立成一个文件（calculator.py），之后可以替换成任意高级公式

**【模块四：前端交互模块】**
用户需要一个看得见的界面来答题和看结果。我们用 Vue 3 做了三个页面：首页（项目介绍 + 开始按钮）、答题页（动态加载题目，4 阶段逐步推进）、结果页（显示人格类型 + 五维雷达图 + 深度分析报告）。
- 解决了什么问题：可交互的图形界面，不是黑乎乎的控制台
- 可行性：Vue 我们之前没学过，这部分主要靠 AI 生成框架再自己调样式。ECharts 是现成的图表库，接入比较简单

**【模块五：结果展示模块】**
结果不能只给四个字母。我们在结果页显示了：大号渐变的人格字母、人格标题和详细描述、五维雷达图（ECharts）、一段 500 字左右的个性化深度分析报告。16 种人格各有一套单独的配色方案，背景色也会根据答题过程累积的权重变化。
- 解决了什么问题：让结果看起来是有温度的分析，不是冷冰冰的标签
- 可行性：ECharts 雷达图和 CSS 渐变都是现成的技术

## 二、设计

### 1. 概要设计

**运行环境：** 后端运行在 Windows + Python 3，前端运行在浏览器（Chrome/Edge），前后端通过 HTTP 通信。

**开发语言和框架：**
- 后端：Python 3，Flask 框架
- 前端：JavaScript，Vue 3 框架 + Naive UI 组件库
- 图表：ECharts 5
- HTTP 请求：Axios

**程序结构：**

前端三个页面通过 Vue Router 切换：
```
首页 (/) → 答题页 (/test) → 结果页 (/result)
          ↓                  ↓
   每答完5题提交一次    显示人格+雷达图+分析报告
   重复 4 次
```

后端 API 层（4 个端点）：
```
POST /api/session/create    → 创建新会话，返回第一组题
GET  /api/pool/<id>         → 获取指定题池的 5 道题
POST /api/pool/<id>/submit  → 提交 5 个答案，返回路由结果 + 当前颜色
POST /api/result            → 最终结算，返回人格 + 雷达图 + 分析报告
```

后端逻辑层（3 个模块各自独立）：
```
app.py        ← API 层，只会把请求分发给两个引擎，不做任何逻辑判断
engine.py     ← 路由引擎：只负责 L/R 多数决路由，不知道权重的存在
calculator.py ← 计算器：只负责五维雷达图 + 颜色计算，不知道路由的存在
reports.json  ← 16 份分析报告文本，跟代码完全解耦
```

**数据流（一个用户的完整流程）：**
```
1. 用户打开首页 → 点击开始测试
2. 前端调 createSession → 后端创建会话，返回 session_id 和第一组题
3. 前端调 getPool(1) → 后端返回 Pool 1 的 5 道题（不含权重）
4. 用户逐题作答（4 选 1）
5. 答完 5 题后调 submitPool(1) → 后端：
   a. engine.py 统计 L/R 多数决 → 记录路径
   b. calculator.py 累加 5 个选项的权重 → 返回当前颜色
   c. 查路由表 → 返回下一题池编号
6. 前端加载下一题池 → 重复步骤 3-5 共 4 次
7. 最后一次提交后 → 前端调 getResult → 后端：
   a. engine.py 从路径生成 4 位人格码
   b. calculator.py 从权重生成五维雷达图数据 + 颜色
   c. 从 reports.json 读取对应人格的分析报告
   d. 拼在一起返回给前端
8. 前端显示结果页：人格字母 + 标题 + 描述 + 雷达图 + 分析报告 + 再测按钮
```

**核心数据：**

每个用户的会话存着这些信息：
```
session = {
    "path": ["L", "R", "L", "L"],    ← 路由路径，4步
    "pool_history": [1, 2, 4, 8],    ← 走过的题池
    "current_pool": 8,               ← 当前在哪
    "weights": [12, 8, 15, 6, 10],   ← 五维累加值
    "done": true                      ← 是否已完成
}
```

题库（pools.json）的结构：
```
15 个题池（编号 1-15），每个包含：
  - stage（1-4）
  - name、description
  - route_map（如 {"L": 2, "R": 3}）
  - 5 道题，每道题包含：
    - id、text
    - 4 个选项（A/B/C/D），每个选项包含：
      - text、route（L或R）、weights（5个数字）
```

### 2. 详细设计

**核心算法一：二分树多数决路由**

```
每 5 题为一组。每题 4 个选项，A/B 指向左（L），C/D 指向右（R）。
5 题答完之后，统计哪边选的多：
  选 L 多 → 往左走 → 进入左分支的下一个题池
  选 R 多 → 往右走 → 进入右分支的下一个题池

一共走 4 层：
  Stage 1（1个池）：决定 I（外放）或 R（内收）
  Stage 2（2个池）：决定 P（参与）或 S（观察），按 I/R 分流
  Stage 3（4个池）：决定 A（分析）或 C（语境），按前两阶段分流
  Stage 4（8个池）：决定 D（深度）或 F（碎片），按前三阶段分流

4 步走完 → 路径转译成 4 位人格码。
   如 ["L","R","L","L"] → 查映射表 → I + S + A + D = "ISAD"（结构冲击者）
```

为什么不同路径不用相同的题池？
每个阶段的题池虽然测的是同一个维度，但因为用户的前置路径不同，直接用同一套题目会出问题。比如一个外放型的人和一个内收型的人，他们在"参与还是观察"这个维度上的表现方式完全不同。如果用同一道题去问他们，结果会不准。所以我们为不同路径准备了不同的题池，题目场景不同，但判定标准一样。

**核心算法二：五维权重累加（与路由完全独立）**

每道题每个选项背后有一个 `weights` 数组，长度 5，分别对应：
```
[精神续航, 社交磁场, 秩序洁癖, 压力解压阀, 脑力劳作]
```
每个位置 0-2 分。用户选一个选项，这 5 个数字就加到这个人的会话权重里。用户完全感知不到这个机制。20 题走完后，累加器里的数据被归一化到 0-100 范围，渲染成五维雷达图。

这个算法的特点是：人格判定和五维图在代码层面完全不相关（一个走路由，一个累权重），但在心理层面它们是先验相关的——一个人的底层心理动机会影响他的行为模式。这个设计让那些"行为与动机产生错位"的离群值变得特别有趣。

## 三、实现

### 1. 后端 API 层（app.py）

后端用 Flask 写了 4 个接口。

**获取题池：** 根据题池编号从 JSON 里读出题目，过滤掉权重和路由信息后返回。
```python
@app.route("/api/pool/<int:pool_id>")
def get_pool(pool_id):
    questions = engine.get_pool_questions(pool_id)
    return jsonify({"id": pool_id, "questions": questions, ...})
```

**提交答案：** 接收 5 个答案，交给 engine 做路由，交给 calculator 累权重。
```python
@app.route("/api/pool/<int:pool_id>/submit", methods=["POST"])
def submit_pool(pool_id):
    # engine 做路由
    result = engine.submit_pool_answers(sess, pool_id, answers)
    # calculator 累权重
    for i, ans in enumerate(answers):
        opt = questions[i]["options"][ans]
        calc.accumulate(sess["weights"], opt["weights"])
    return jsonify({"next_pool": ..., "is_final": ..., "color": ...})
```

**提示词：** "帮我写一个 Flask 后端，有 4 个接口。一个是创建会话，一个是获取题目，一个是提交答案并路由，一个是最终结算。"

### 2. 路由引擎（engine.py）

路由引擎只做一件事：统计 L/R 次数，根据多数决路由。

```python
def submit_pool_answers(self, session, pool_id, answers):
    route_counts = {"L": 0, "R": 0}
    for ans in answers:
        label = questions[i]["options"][ans]["route"]
        route_counts[label] += 1

    result = "L" if route_counts["L"] >= route_counts["R"] else "R"
    next_pool = pool["route_map"][result]
    return {"next_pool": next_pool, "route_result": result}
```

### 3. 权重计算器（calculator.py）

独立于路由引擎，可放任意高级公式。

```python
class WeightCalculator:
    def compute_radar(self, state):
        """五维雷达图归一化（以实际最大值为基准）"""
        base = max(20, max(state[:5]))
        return [{"name": self.dim_names[i], "value": min(100, int(state[i]/base*100))}
                for i in range(5)]

    def compute_color(self, state):
        """从权重映射到 RGB 颜色"""
        r = min(55, 8 + state[3] * 0.7)
        g = min(45, 6 + state[1] * 0.5)
        b = min(55, 10 + state[4] * 0.6)
        return f"rgb({r},{g},{b})"

    def compute_all(self, state, stage):
        return {"radar": self.compute_radar(state),
                "color": self.compute_color(state),
                "gradient": self.compute_color_gradient(state, stage)}
```

**提示词：** "把权重计算从路由引擎里分离出来，单独放到一个文件里。它负责五维雷达图的计算和背景颜色的计算，跟路由逻辑完全无关。"

### 运行结果分析

系统经过多轮测试，16 种可能的路径全部正确路由，每种路径输出的 4 位人格码和对应的人格标题、描述、分析报告都正确匹配。五维雷达图的数值随答题选项不同而正常变化，背景色也按阶段和权重逐步过渡。

### 遇到的问题

**问题 1：中文引号导致编译错误**
在人格描述里用了中文双引号（"后勤保障"），Python 把它当成了字符串结束符，报了 82 个级联错误。后来把所有中文引用改成了「」角括号。

**问题 2：会话丢失导致路由不累积**
最初用 Flask 的 cookie session 保存用户状态，但前端没把 cookie 带回来，每次请求都创建新会话，路径数据全部丢失。后来改为前端显式传递 session_id，后端按 id 查找对应的会话数据，问题解决。

**问题 3：选项倾向太明显**
初版题目有些选项一眼就能看出选哪个会走哪个方向。逐池审阅后修改了倾向太明显的选项，让 4 个选项更均衡。

**问题 4：首版用了 Java Spring Boot 后弃用**
最初尝试用 Java Spring Boot + Vue 做，但环境和配置极其麻烦（要装 JDK 17、Maven、配 CORS），花了很多时间才跑通。后来换成了 Python Flask，两个文件就搞定了后端，开发效率大幅提升。
