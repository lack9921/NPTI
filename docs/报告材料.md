## 一、构思

### 1. 项目背景和任务目标

**（这里直接复制任务书里题目二的主线任务+额外挑战，把 "题目二" 改成 "我们的选题" 即可）**

> 任务书原文：
> 题目二：MBTI测试系统
> 主线：实现包含题库、测试交互、分数计算的系统。需通过特定算法计算出人格类型并输出详细解析。强烈鼓励自创测评体系（如SBTI），定义全新的维度、权重，并在报告中详细阐述。
> 额外挑战：使用图形化界面进行交互，并在测试结束后使用 Echarts 生成包含交互图表的个人专属报告。

**我们的目标：** 做一个基于 **二分树动态路由** 的 NPFJ 人格测评系统，用户每答 5 题，系统根据选择方向自动路由到下一组题。最终输出 4 位人格类型 + 五维心理雷达图。

### 2. 需求分析

要完成这个系统，需要实现以下几个模块：

**① 题库模块**
- 功能：存储所有题目，每道题有题目文字、四个选项，每个选项标注了"方向"和"权重值"
- 解决什么问题：题目和代码分离，改题不改代码
- 实现方式：题目存在 JSON 文件里，后端启动时读取
- 技术可行性：JSON 是标准格式，Python 自带 json 库可以读取，没有任何难度

**② 路由引擎模块**
- 功能：接收用户的一组答案（5 题），统计方向，决定走到下一个题池
- 解决什么问题：实现"选什么 = 看到什么不同"的动态测试流程
- 实现方式：每个题池有一个路由表，比如 {"L": 2, "R": 3}，选 L 多就走 2 号池，选 R 多就走 3 号池
- 技术可行性：if-else 和字典查找，大一学的就够用

**③ 五维权重累加器**
- 功能：每道题的选项背后有 5 个心理维度的权重值，用户答题时不知不觉累加
- 解决什么问题：提供一个独立于人格判定的心理倾向分析（雷达图）
- 实现方式：每次提交答案时，把每个选项的权重数组加到 session 的累加器里
- 技术可行性：数组加法，没难度

**④ 前端交互模块**
- 功能：显示题目、接收选择、提交答案、展示结果
- 解决什么问题：用户需要看得见的界面
- 实现方式：Vue 3 前端，三个页面（首页/答题页/结果页）
- 技术可行性：Vue 是前端框架，需要额外学。我们原来没学过，这部分主要靠 AI 生成和调试。

**⑤ 结果展示模块**
- 功能：显示人格类型、描述、五维雷达图
- 解决什么问题：让用户看到可视化的分析结果
- 实现方式：ECharts 画雷达图，16 种人格每种一套配色
- 技术可行性：ECharts 是现成的 JS 图表库，接入比较简单

## 二、设计

### 1. 概要设计

**运行平台：** Windows + 浏览器（前后端分离，后端跑 Flask 服务器，前端跑在浏览器里）

**开发语言：**
- 后端：Python 3（Flask 框架）
- 前端：JavaScript（Vue 3 框架）

**依赖库：**
- 后端：flask、flask-cors
- 前端：vue、vue-router、axios、echarts

**程序模块结构：**

```
┌─────────────────────────────────────────────┐
│               前端（Vue 3）                   │
│  HomeView  →  TestView  →  ResultView        │
│  （首页）     （答题页）      （结果页）        │
│                ↓ axios HTTP                   │
│           /api/pool/1  /api/pool/1/submit     │
│           /api/session/create  /api/result     │
└─────────────────┬───────────────────────────┘
                  │
┌─────────────────▼───────────────────────────┐
│              后端（Python Flask）             │
│  app.py  ←  API 层（4 个端点）               │
│    ↓                                         │
│  engine.py  ← 路由引擎 + 权重累加器           │
│    ↓                                         │
│  pools.json  ← 题库（15 个题池 × 5 题）      │
└─────────────────────────────────────────────┘
```

**核心数据结构和变量：**

后端 Session（每个用户一个会话）：
```
session = {
    "path": ["L", "R", "L", "L"],    ← 走过的路由方向
    "pool_history": [1, 2, 4, 8],    ← 经过的题池编号
    "current_pool": 8,               ← 当前在哪
    "weights": [12, 8, 15, 6, 10],   ← 五维累加值
    "done": false                     ← 是否已完成
}
```

题库结构（pools.json）：
```
每个题池包含：编号、阶段、名称、描述、路由表、5道题
每道题包含：题号、题目文字、4个选项
每个选项包含：选项文字、路由方向(L/R)、五维权重[5个整数]
```

**模块调用关系：**
1. 用户打开首页 → 前端创建 session → 后端返回 session_id 和第一组题
2. 用户答题 → 前端提交 5 个答案 → 后端路由引擎统计方向 → 累加权重 → 返回下一组题
3. 重复第 2 步共 4 次 → 最后一组提交后后端返回完成标识
4. 前端调结算接口 → 后端把路径转成 4 位人格码 → 查人格描述 → 返回结果 + 五维数据
5. 前端用 ECharts 渲染雷达图展示结果

### 2. 详细设计

**核心算法：二分树多数决路由**

```
每 5 题一组，每题 4 个选项。
A/B 选项指向 L 方向，C/D 选项指向 R 方向。
5 题中选 L 多 → 往左分支走
5 题中选 R 多 → 往右分支走

以 Stage 1（能量表达）为例：
用户答完 Pool 1 的 5 题。
如果选 A/B（L方向）共 4 次以上 → 判定为 I（外放型）
如果选 C/D（R方向）共 3 次以上 → 判定为 R（内收型）
然后路由到下一阶段的对应题池
```

算法优缺点：
- 优点：简单易懂，实现起来就是统计次数，不会出错
- 缺点：如果刚好 3:2 或 2:3，那少数的那两题完全没有影响。但对于人格测试来说，我们觉得看主要倾向就够了，小偏差忽略问题不大

### 2. 路由引擎（engine.py）

**功能：** 接收 5 个答案，统计 L/R 次数，累加权重，返回下一题池。

关键代码：

```python
def submit_pool_answers(self, session, pool_id, answers):
    # 统计 L/R 方向
    route_counts = {"L": 0, "R": 0}
    for i, ans in enumerate(answers):
        opt = questions[i]["options"][ans]
        label = opt["route"]
        route_counts[label] += 1
        # 累加五维权重
        for j in range(5):
            session["weights"][j] += opt["weights"][j]

    # 多数决
```

### 运行结果分析

经过测试，系统可以正常运行。用户从首页进入后，依次回答 4 个阶段各 5 道题，总共 20 题，最终能正确输出 16 种人格类型之一并显示对应的雷达图。四种不同的答题路径测试结果均符合预期。

### 遇到的问题

**问题 1：中文引号导致 Python 报错**
在人格描述里有类似"后勤保障"这样的中文双引号，Python 把它当成了字符串结束符，导致编译错误。后来把所有中文引用改成了「」角括号解决。

**问题 2：前后端联调跨域问题**
Flask 后端默认不允许跨域请求，前端调后端时会报错。加了 Flask-CORS 和 CORS 配置后解决。

**问题 3：题库选项倾向太明显**
初版的题目有些选项一眼就能看出选哪个会走哪个方向，后来逐池修改，让四个选项更均衡，用户需要认真考虑才能做出选择。
