# MBTI 测试系统 — 开发实战手册

> **选题**：题目二 · MBTI 测试系统（自创 SBTI 测评体系）
> **技术栈**：Java Spring Boot（后端）+ Vue 3（前端）
> **截止日期**：2026 年 7 月 5 日（周日）
> **团队**：5 人（3 代码 + 1 报告 + 1 视频）

---

## 一、开干前的共识

### 1.1 核心原则

- **每个人独立完成自己的模块**，不需要等别人。前端可以先写假数据把页面画好，后端可以先写 API 用 Postman 测好。
- **AI 是主力写手**，但每段代码必须看懂它在干什么。给 AI 的提示词要细到"函数名、输入参数类型、返回值格式"这个级别。
- **不要追求完美架构**，能跑通、演示视频好看就是胜利。代码质量不重要，演示效果最重要。

### 1.2 时间窗口

| 时间段 | 做什么 |
|--------|--------|
| 7/3（周五）白天 | 搭环境 → 定接口 → 各写各的核心功能 |
| 7/3（周五）晚上 | 前后端联调 → 跑通主流程 |
| 7/4（周六）全天 | 修 Bug → 美化 UI → 加 Echarts → 报告 + 视频同步推进 |
| 7/5（周日） | 最终打包 → 录视频 → 提交 |

### 2.2 VS Code Live Share 配置（非常重要！）

Live Share 让三个人可以在同一个 VS Code 里**实时协同编辑同一个项目**，不用折腾 Git 合并。

**使用流程：**

1. **全员**在 VS Code 扩展商店搜索安装 `Live Share`（微软官方出品，蓝色图标）
2. **任意一人**（建议 Coder 2，因为后端先搭好）：
   - 点击 VS Code 左侧的 Live Share 图标
   - 点击 "Start Collaboration Session"
   - 会生成一个分享链接，复制到群里
3. **另外两人**点击链接加入，就能看到同一个项目文件树
4. **协作技巧**：
   - 每个人可以用鼠标点一下自己要改的文件，**其他人能看到你的光标在哪儿**
   - 右上角可以切换 "跟随模式"（Follow Mode），能实时看别人敲代码
   - 终端（Terminal）是**各自独立的**，可以各自跑自己的命令
   - 谁改了什么文件，文件名旁边会显示他的头像

**⚠️ 注意：** Live Share 适合**三个人一起写同一个项目**。前端和后端是独立的文件夹，三个人可以在同一个 VS Code 窗口里各写各的文件夹，互不干扰。

## 三、分工（重新定义）

### 3.1 核心思想

**不按技术层分（前端/后端/胶水），而是按端到端功能模块分。**

每个人独立负责一整个功能模块，写完就能自己测通，不需要等别人。

### 3.2 三位 Coder 的分工

```
项目根目录/
├── npti-frontend/          ← Coder 1 全权负责 (Vue 3 前端)
│   └── src/
│       ├── views/          ← 页面文件（首页/答题页/结果页）
│       ├── components/     ← 组件文件（雷达图等）
│       ├── router/         ← 路由配置
│       ├── api/            ← 调后端的请求封装
│       └── assets/         ← 全局样式
│
├── npti-backend/           ← Coder 2 全权负责 (Spring Boot 后端)
│   └── src/main/java/com/npti/
│       ├── controller/     ← API 接口
│       ├── service/        ← 算分逻辑（核心）
│       ├── dto/            ← 数据结构（接口契约）
│       └── config/         ← 跨域配置
│
├── docs/                   ← 文档
│   └── 开发实战手册.md      ← 完整开发指南
│
└── .gitignore
    └── echarts-demo/       ← Echarts 测试
```

### Step 5：联调（后端跑通 → 前端接入）

**Coder 2 先做：**
1. 跑起来 `mvnw spring-boot:run`
2. 用浏览器访问 `http://localhost:8080/api/test/submit`（或者用 Postman 发 POST 请求）
3. 确认能收到 JSON 返回

**Coder 3 接着做：**
1. 确认 Coder 2 的后端跑通
2. 让 Coder 1 把前端 `api/request.js` 里的后端地址从 `localhost` 改成 Coder 2 电脑的局域网 IP
   - 查 IP 命令（Windows）：`ipconfig` → 找到 `IPv4 地址`
   - 查 IP 命令（Mac/Linux）：`ifconfig` 或 `ip addr`
3. 让 Coder 1 去掉前端假数据，换成真接口请求
4. 前端点一下"提交"按钮，看能不能返回数据

**如果联调不通，大概率是这几个问题：**
| 问题 | 解决方案 |
|------|---------|
| 跨域报错 | 检查 Coder 2 的后端有没有加 `@CrossOrigin` 或 CorsConfig |
| 前端调不通后端 IP | 检查两个电脑是不是连同一个 WiFi，ping 一下对方 IP |
| 数据格式不匹配 | 检查前后端 JSON 字段名是否一致（大小写敏感） |
| 后端端口被占用 | 改 `application.yml` 里的 `server.port` 为 8081 |

### Step 6：加上自创 SBTI 维度（加分项）

如果基础 MBTI 跑通了，让 Coder 2 在 Service 里加一个**第二套算分逻辑**：

返回结果里多加一个 `sbtiType` 字段。这就是报告里吹的"**自创测评体系**"，老师看到直接加分。

## 五、打包部署（最后一步）

### 5.1 前端打包

```bash
cd npti-frontend
npm run build

会在 `dist/` 文件夹生成静态文件。

### 5.2 整合方案

**最简单方案（推荐）：** 直接将 `dist/` 文件夹放进后端的 `src/main/resources/static/` 目录下，然后后端打成 jar 包，一个命令跑起来就能访问所有页面。

请告诉我如何把 dist/ 文件夹集成到 Spring Boot 项目中，
以及如何把整个项目打成可执行的 jar 包。
```

## 六、避坑指南

### 🚫 不要做的事

1. **不要现学 Git 分支合并** —— 你们三个人用 Live Share 就够了，Git 只用来最后 push 到仓库
2. **不要在联调前改接口格式** —— 定好的 JSON 字段名不要改，改之前群里说一声
3. **不要让 AI 一次性生成整个项目** —— 粒度要小，一次生成一个文件/一个组件
4. **不要等代码写完才开始拍视频和写报告**
5. **不要在最后一刻才打包测试**

### ✅ 要做的事

1. **每跑通一个功能就截图** —— 报告同学时刻准备截屏
2. **每写一个文件就检查能不能看懂** —— 看不懂的地方问 AI："请解释这段代码每一行在做什么"
3. **遇到 Bug 先自己看 5 分钟**，看不懂再让 AI 修
4. **视频要控制在 2-5 分钟**，太长老师不看
5. **最终压缩包里的文件名要规范：组号-题目二.zip**
